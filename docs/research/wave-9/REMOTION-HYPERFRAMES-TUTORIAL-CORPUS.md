## 2. Cross-creator consensus (what *most* recommend)

These are points where 3+ independent creators converge — the strongest signals.

1. **The agent (Claude Code) is the orchestrator, and you do NOT need to code.** Every single creator frames it this way. The human's job is prompting, curation, and taste — not writing React/HTML. (All 11.)

2. **Install the engine's official "skill" / best-practices file and let it compound.** Remotion ships `/remotion best-practices`; Hyperframes ships `/hyperframes` + writes `agents.md`/`CLAUDE.md`. Creators stress invoking it at session start and that these files "get smarter over time." (Brooke, Fazt, jonyl, Hans, Charlie — both engines.)

3. **Work inside a dedicated per-video / per-project folder, and `/clear` or start fresh between phases.** Stops the agent searching the whole machine, saves tokens, keeps context clean. (Charlie, Fazt, Brooke — matches **Nate Herk's** token-discipline point exactly.)

4. **Iterate, don't one-shot.** Universally: first render is rough; quality comes from a few feedback passes. Remotion camp does it via **incremental step-by-step prompts** (Fazt explicitly says the Remotion authors recommend this); Hyperframes camp via the **visual studio editor + "make bigger structural changes"** (Charlie). (Fazt, Charlie, Hans, Brooke, jonyl.)

5. **Word-level transcript/timestamps drive overlay sync — use Whisper.** Overlays/captions must land on the spoken word. Brooke runs **local Whisper**; Hans prompts "use Whisper to extract the script with precision"; Charlie has a dedicated **scene-anchor agent** for timestamp↔overlay matching. (Brooke, Hans, Charlie — **identical to Nate Herk's single most-repeated principle**, and to our existing faster-whisper word timings.)

6. **Build a reusable brand/style system and reusable components.** Brand kit doc (hex, typography), reusable hook/CTA components, per-type design philosophy file. Design once, swap data to mass-produce. (Charlie's `brand-kit.md`, Brooke's reusable components, Hans's reusable templates, Fazt's swap-the-data — matches **Nate Herk's** "lesson-design.md / design philosophy" idea.)

7. **Plan/review before the expensive step.** A scene-by-scene plan (anchor word, timing, palette) reviewed before rendering. (jonyl's prompt-the-prompt scene breakdown, Hans's pre-made plan, Matt's script+shot-list gate, Charlie's QA agent — matches **Nate Herk's** "plan beats before rendering.")

8. **Renders are slow and RAM/cost-heavy on a local machine — budget for it.** 10–30 min renders (Charlie), ~10–13 min (jonyl), token cost is real. (Charlie, jonyl, Matt, + **Nate Herk's** ~238K-token / RAM-thrash warnings.) Directly relevant to our local-only M-series constraint.

9. **The generated output still "looks AI-generated"; taste is the differentiator.** Designers spot it (Cmd+AI); Matt warns the whole genre will commoditize. The moat is human taste + a recurring brand/character. (Cmd+AI, Matt, + **Nate Herk's** "taste is the moat.")

## 4. Agreement / disagreement with Nate Herk

Nate Herk is our anchor (he runs the *exact* Claude Code + Hyperframes + video-use stack). How the broader corpus stacks up against his §3 recommendations:

**Strong agreement (corpus reinforces Nate):**
- **Word-timestamp-driven sync via Whisper** — Nate's #1 principle; independently echoed by Brooke (local Whisper), Hans (Whisper subtitles), Charlie (scene-anchor agent). **High confidence — adopt.**
- **Plan beats before rendering** — Nate's "planning stage is very important"; echoed by jonyl (prompt-the-prompt scene plan), Hans (pre-made plan), Matt (script+shot-list gate). **Adopt.**
- **Per-type design-philosophy file + reusable components** — Nate's "lesson-design.md"; echoed by Charlie (`brand-kit.md`), Brooke (reusable components), Hans/Fazt (reusable templates). **Adopt.**
- **Screenshot self-verification** — Nate's screenshot-verification trick; **Hans's "screenshot the error and send the image"** is the same mechanism for debugging, and Charlie's QA agent enforces a 95/100 gate. **Adopt as a gate.** (We already have `scripts/extract-keyframes.py`.)
- **Token/RAM discipline, `/clear` between phases, one render at a time** — Nate's explicit warnings; echoed by Charlie (dedicated folder to save tokens), Fazt (`/clear` between clips), and the slow-render reports (jonyl 10–13 min, Charlie 10–30 min). **Adopt — critical for our local-only box.**
- **Taste is the moat** — Nate's exact phrase; echoed by Cmd+AI ("looks AI-generated") and Matt ("commoditizes — need a character/brand moat"). **Cultural agreement.**

**Disagreement / nuance the corpus adds (where Nate is NOT the whole story):**
- **Engine choice is more contested than Nate implies.** Nate prefers Hyperframes "because we just like it better." But the corpus is **split ~5 Hyperframes / 4 Remotion / 2 both**, and the Remotion camp (esp. Hans) makes a strong **determinism/consistency** argument Nate doesn't emphasize. NoCodeAbode even preferred Remotion's *look* on the same prompt. → For our bake-off, **Nate's "Hyperframes looks better" is a real but not consensus view.**
- **video-use vs manual trim.** Nate uses video-use; **Charlie explicitly distrusts agent auto-trim** and prefers Tella. → Lean toward keeping trim outside the AI loop.
- **The Remotion `/remotion best-practices` skill is excellent and underrated.** Nate's videos skew Hyperframes; the Spanish trio (Fazt/jonyl/Hans) shows the Remotion skill is mature, multi-agent (works with GPT/Gemini/Antigravity), and the **incremental step-by-step prompting** style is the officially-recommended Remotion workflow — a concrete technique Nate's Hyperframes-centric coverage doesn't surface.
- **Reliability favors Remotion.** Nate himself conceded Hyperframes previews were flaky while Remotion's never were — and the corpus gives no contradicting evidence strong enough to overturn that for a reliability-sensitive local pipeline.

## 5. What we should adopt (recommendations for THIS repo)

Given our constraints (local-only M-series, Spanish business/educational content, existing working Remotion stack + Hyperframes installed for bake-off, faster-whisper already producing word timings):

**Engine decision (bake-off input):**
1. **Keep Remotion as the reliable default**, consistent with Nate's reliability concession and the Remotion camp's determinism argument (Hans). Our pipeline's job is **deterministic branded overlays/charts/captions** — exactly Remotion's strength, and consistency is a feature, not a bug, for repeatable templated output (`graficos-de-video`, monthly-style reports).
2. **Steal the Hyperframes *aesthetic*, not necessarily the engine** — the "more sophisticated/cohesive look" is mostly **liquid-glass flank cards, dual-HUD overlays, and the 11 laws of motion design** (Alex Hitt). These are pure CSS `backdrop-filter` + transforms, fully buildable as Remotion components (matches Nate's §4.4 conclusion). Encode the **11 laws** ("one idea per beat, cut fast," cap reveals ≤0.6 s, breathing beats, motion blur on fast transitions) into our composition guidelines / brand design doc.
3. **Avoid GSAP inside Remotion** — use native `interpolate`/`spring` to sidestep the wall-clock black-frame bug (DIY Smart Code, Alex Hitt). If we ever want GSAP-style timelines, that's the one concrete reason to render that scene through Hyperframes instead.

**Workflow practices to adopt (consensus-backed):**
4. **Word-timestamp-driven overlay sync** — extend our existing faster-whisper word timings to drive overlay `enter` times, not just captions (consensus §2.5; we don't need their ElevenLabs/OpenAI dependency).
5. **Emit a reviewable scene-plan before rendering** — anchor word, timing, layout, overlay kind, palette — cheap to review, expensive to redo (consensus §2.7).
6. **Per-template design-philosophy markdown** (`<template>-design.md`) layered on `brand/config.json` — so style compounds (consensus §2.6).
7. **Screenshot-verification gate before full render** — render keyframes, have the agent confirm framing (face not covered, text in-frame, nothing blurred); and adopt **Hans's "screenshot the error → send image"** debugging loop (consensus §2.5 + Nate). Reuse `scripts/extract-keyframes.py`.
8. **Token/RAM discipline** — dedicated per-video folder, `/clear` between trim→plan→render phases, render ONE composition at a time (consensus §2.3/2.8; critical on our 8GB-class local box).
9. **Keep trimming outside the heavy AI loop** — pre-clean obvious mistakes or use a Tella-style tool; don't trust agent auto-cut for final timing (Charlie's caution + Nate's "pre-trim raw footage").
10. **Reusable component library + screenshot-driven style matching** — build hook/CTA/lower-third/flank-card as reusable Remotion components; when matching a look, **feed Claude a reference screenshot** of the target style (Brooke's key tip).

**Out of scope but noted:** generative-clip B-roll (Higgsfield/Veo, Matt's pipeline) is a *complement*, not a replacement — could feed FFmpeg-composited B-roll under our deterministic Remotion overlays later, but not part of the engine decision.
